서비스 팀은 변경 사항을 사용자에게 알리기 위해 변경 로그를 사용할 수 있습니다.

변경 로그는 서비스 페이지를 열 때 한 번 표시됩니다. 변경 로그 페이지를 다시 열 수 있는 옵션은 제공하지 않습니다. 필요한 경우 독립적으로 구현하십시오.

변경 로그는 모달 창에 표시되므로 모달 창을 닫고 크기를 조절하는 규칙을 따릅니다.

## 요소

변경 로그는 시간순으로 정렬된 게시물이 있는 M 크기의 모달 창입니다. 각 게시물에는 다음이 포함됩니다.

1. 게시 날짜
2. "새로운" [레이블](https://gravity-ui.com/design/guides/label) (선택 사항)
3. 게시물 헤더
4. 게시물 이미지 (선택 사항)
5. 게시물 텍스트
6. 작업 [버튼](https://gravity-ui.com/design/guides/button) (선택 사항).

![Elements](/static/images/design/ChangelogDialog/ChangelogDialog1.png)

### 게시 날짜

게시 날짜를 통해 사용자는 서비스에 특정 변경이 발생한 시점을 이해할 수 있습니다. 기본적으로 날짜 형식은 DD Mon YYYY입니다. 서비스 로케일에 맞게 다른 날짜 형식을 사용할 수도 있습니다.

### 레이블

새로운 게시물에는 "새로운" 레이블을 사용하십시오. 서비스는 어떤 게시물을 새로운 것으로 간주하고 이 레이블로 표시할지 결정합니다. 텍스트에서는 "새로운"과 다른 문구를 사용할 수 있습니다.

### 게시물 헤더

게시물 헤더는 서비스 변경 사항에 대한 간략한 설명을 제공하는 데 사용됩니다. 헤더는 길이에 제한이 없습니다. 다음을 권장하지 않습니다.

- 한 줄 이상을 차지하는 헤더(70자 이상)는 가독성을 떨어뜨리므로 사용하지 마십시오.
- 뉴스 텍스트를 헤더에 복사하지 마십시오.

헤더는 text-subheader-3 스타일을 사용합니다. 다른 텍스트 스타일은 헤더에 제공되지 않습니다.

### 이미지

이미지는 선택 사항이며 516x240px의 고정 크기입니다. JPG 및 PNG 파일을 이미지로 사용할 수 있습니다 (GIF 이미지 및 비디오 파일은 지원되지 않습니다). 게시물 주제를 설명해야 할 때 이미지를 사용하십시오. 하나의 이미지는 밝은 테마와 어두운 테마 모두에 사용됩니다.

516x240px보다 큰 이미지를 업로드하면 가장자리가 늘어나고 가운데 정렬됩니다.

Retina 디스플레이를 고려하여 각 면이 두 배로 확장된 이미지, 즉 1032x480px를 업로드하는 것이 좋습니다.

### 게시물 텍스트

게시물 텍스트는 이미지 뒤에 오며 길이에 제한이 없습니다. 다음을 권장하지 않습니다.

- 매우 긴 텍스트(500자 이상)는 사용하지 마십시오. 텍스트를 권장 길이로 줄일 수 없는 경우, 텍스트 내에 문서 링크를 추가하거나 텍스트 뒤에 전체 텍스트로 이동하는 버튼을 제공할 수 있습니다.
- 부제목을 만들지 마십시오. 권장 텍스트 길이가 간결하므로 부제목을 사용하여 텍스트를 섹션으로 나눌 필요가 없습니다.

텍스트는 'text-body-1' 스타일을 사용합니다. 다른 텍스트 스타일은 제공되지 않지만, 글머리 기호 목록 및 번호 매기기 목록, 취소선, 밑줄, 굵게, 기울임꼴과 같은 서식을 추가할 수 있습니다.

### 작업 버튼

작업 버튼은 선택 사항이며 게시물 텍스트 아래에 있습니다. 게시물의 전체 텍스트로 이동하는 데 사용하십시오.

M 크기의 버튼만 지원됩니다. 버튼은 아이콘이 있거나 없을 수 있으며 어떤 유형이든 가능합니다. 단일 버튼 유형을 사용하십시오.

여러 개의 버튼은 지원되지 않습니다.

### 기타 요소

또한 모달 창 헤더 아래에 "전체 변경 로그 보기" 링크를 변경 로그 페이지(별도의 서비스 페이지, 스토리북 또는 외부 문서 서비스일 수 있음)에 추가할 수 있습니다.

## 사용 권장 사항

변경 로그 사용:

- 서비스의 중요한 변경 사항을 사용자에게 알려야 할 때.
- 사용자가 사소한 변경 사항을 파악하도록 해야 할 때. 이 경우 변경 로그를 주기적으로 표시하고 최신 변경 로그 항목 이후의 모든 사소한 변경 사항에 대해 알리는 것을 고려하십시오.
- 서비스의 변경 사항을 사용자에게 상기시키고 이전 릴리스에 대한 인식을 높여야 할 때.

다음은 권장하지 않습니다.

- 변경 로그를 사용하여 사용자를 온보딩하는 것.
- 변경 로그를 사용하여 서비스의 모든 사소한 변경 사항을 사용자에게 알리는 것.
